스펙 작성 전 사전 단계 계획

스펙을 적기 전에 무엇을 먼저 해야 하는가

베스트 프랙티스 리서치와 1단계 첫 출시판(법령 의무 자동화 어플) 맥락에 맞춘 사전 작업 6주 제안.

대상
1단계 첫 출시판 — 손해사정서 자동 작성 + 피해자 전자 확인 + 보험사 교부 기록
입력
docs/reports/painpoints.html, docs/reports/service-direction.html, 별첨1·2 손해사정 보고서, 사업계획서
결정 대상
(1) 사전 단계 베스트 프랙티스 6가지, (2) 우리 프로젝트 6주 일정, (3) 단계별 산출물, (4) 피해야 할 함정
출처
2026년 발표된 스펙 기반 개발(SDD) 가이드, 도메인 주도 설계 자료, 시범 운영(MVP) 가이드, 기술 결정 문서(RFC) 자료. 8장 참조.
작성일
2026-05-17

한 줄 결론

스펙(spec/ 폴더)을 적기 전에 도메인 학습 → 사건 흐름 워크숍 → 위험 검증 → 기술 결정 문서 → 뼈대 시스템 배포의 5개 사전 단계를 차례로 거친다. 합계 약 4~6주. 이 사전 단계 없이 스펙부터 적으면 도메인 용어가 빈약해 모호한 스펙이 나오고, 위험한 통합(보험사 교부 기록 채널·전자서명·PDF 자동 작성)이 마지막에 터진다.

핵심 원칙 셋. (가) 스펙은 짧게(1~3쪽), 분명하게, 범위 밖도 명시한다. (나) 위험한 부분부터 얇게 관통해서 작동 가능성을 먼저 확인한다 (예광탄 검증). (다) 배포 가능한 뼈대를 먼저 올린 뒤 그 위에 기능을 채운다 (뼈대 시스템).

1. 왜 사전 단계가 필요한가

스펙 기반 개발(Spec-Driven Development)은 "스펙 → 계획 → 작업 → 구현"의 4단계로 흐른다. 그러나 첫 스펙을 적으려면 그 전에 알아야 할 것이 있다. 이 어플의 경우 특히 그렇다.

  • 도메인 용어가 정착되지 않으면 스펙은 모호해진다. 손해사정서/현장보고서/중간보고서/종결보고서, 안분식·과실상계, 손해사정사 위탁 동의서 등은 외부 시각에서 단어가 비슷해 보이지만 법적 효과와 흐름이 다르다. 용어집과 사건 흐름이 먼저 굳어야 한다.
  • 위험한 통합은 마지막에 터진다. 보험사 교부 기록 채널, 전자서명 법적 유효성, PDF 자동 작성 정확도는 외부 의존이다. 사전 단계에서 얇게 관통해 보지 않으면 1단계 출시 직전에 발견된다.
  • "out of scope"가 정해지지 않으면 스펙이 무한정 부푼다. 1단계 첫 출시판은 한 흐름만 동작하면 된다. 무엇을 넣지 않을지를 먼저 정한다.
  • 뼈대 시스템(Walking Skeleton)을 먼저 배포해 두면 빌드·배포·인증·모니터링 같은 횡단 이슈가 첫날부터 드러난다. 이런 이슈는 기능이 다 만들어진 뒤에 발견하면 비용이 크다.

2. 업계 베스트 프랙티스 6가지 패턴

2026년 시점에 정리된 자료를 종합하면 사전 단계는 다음 6가지 패턴으로 수렴한다. 각 패턴은 우리 프로젝트의 한 단계와 직접 매핑된다.

1 사건 흐름 워크숍 — 이벤트 스토밍(Event Storming)

핵심
도메인 전문가와 개발자가 한자리에 모여 시간 순서대로 일어나는 사건(예: "사고 접수됨", "현장조사 완료됨", "손해사정서 교부됨")을 포스트잇으로 붙여 나열한다. 그 다음 행위자·정책·도메인 묶음(Aggregate)·도메인 경계(Bounded Context)를 식별한다.
왜 필요
도메인 지식이 코드로 들어가기 전에 한 번 정렬된다. 보험사·피해자·사정사 3자의 책임 경계가 명확해진다.
우리 적용
별첨1·2 보고서를 따라가며 사건 흐름을 추출. 손해사정서 교부와 보험사 인증을 별도 사건으로 분리.

2 기술 결정 문서 — RFC / Design Doc

핵심
구현 전에 "무엇을 어떻게 풀 것인가"를 짧은 문서로 적고 동료 리뷰를 받는다. 목표·아키텍처·대안·트레이드오프·결정 사항을 담는다.
왜 필요
의사결정의 근거가 코드 뒤에 숨지 않고 글로 남는다. 1년 뒤 "왜 이렇게 결정했나"를 추적할 수 있다.
우리 적용
1단계 첫 출시판 단위로 RFC 1편. PDF 자동 작성 라이브러리 선택, 전자서명 방식, 교부 기록 채널 선택, 민감정보 암호화 방식을 모두 한 문서에 정리.

3 예광탄 검증 — Tracer Bullet

핵심
가장 위험한 부분을 얇은 수직 단면으로 한 번 관통한다. 프로토타입과 달리 버리지 않고 본 시스템의 시작점이 된다. 정식 기능이 아니라 위험 검증이 목적.
왜 필요
외부 의존(보험사 채널, 전자서명, PDF) 같이 작동 여부 자체가 의문인 부분을 일찍 검증한다.
우리 적용
"빈 손해사정서 → PDF 생성 → 가짜 보험사 메일함으로 송부 → 수신 기록"을 4~5일 안에 한 번 끝까지 통과시킨다.

4 뼈대 시스템 — Walking Skeleton

핵심
최소한의 끝과 끝(end-to-end) 흐름을 운영 환경에 배포 가능한 상태로 만든다. 화면 → API → 데이터베이스 → 인증 → 배포 파이프라인까지 모두 연결되어 있되, 기능은 비어 있다.
왜 필요
빌드·배포·로그·모니터링 같은 횡단 관심사가 첫날부터 드러난다. 이후 모든 기능은 이 뼈대 위에 살이 붙는다.
우리 적용
Next.js 빈 화면 → Spring Boot 빈 API → MySQL 빈 스키마 → Docker 이미지 → 배포까지 1주 안에 한 번 흘러가게 한다. 첫 기능은 "사정사 로그인 → 빈 사건 1건 생성 → DB 저장" 정도.

5 요구사항 문장 패턴 — EARS(Easy Approach to Requirements Syntax)

핵심
요구사항을 다섯 가지 정해진 문장 틀로만 적는다. 모호한 문장이 줄어들고 자동 검증이 쉬워진다. 예: "[사건이 종결로 표시된 경우], 시스템은 [표준 양식대로 손해사정서를 자동 생성]해야 한다."
왜 필요
"잘 처리한다", "적절히 보여 준다" 같은 모호한 문장이 사라진다. 인수 시나리오가 곧바로 도출된다.
우리 적용
스펙 작성 단계에서 채택. 사전 단계에서는 도메인 용어집을 만들어 EARS 문장의 단어들이 이미 정의되어 있도록 한다.

6 범위 밖(out of scope) 명시 + 1~3쪽 짧은 스펙

핵심
스펙은 짧을수록 좋다. 길어지면 쪼갠다. "이 스펙이 다루지 않는 것"을 별도 절로 명시한다.
왜 필요
긴 스펙은 안 읽힌다. 범위 밖이 적히지 않으면 구현 도중 "이것도 해야 하나"가 끊임없이 나온다.
우리 적용
1단계 스펙은 단일 흐름(종결 → 손해사정서 → 전자 확인 → 교부 기록) 한 편으로 시작. 보고서 자동 작성·중복보험 안분 등은 "out of scope"로 명시(2·3단계 이관).

3. 우리 프로젝트 사전 단계 6주 계획

위 6가지 패턴을 단계로 펼치면 다음과 같다. 각 단계는 합격선(다음 단계로 넘어가는 기준)을 갖는다.

0주차

도메인 학습과 자료 수집

소요 1주
활동
  • 별첨1(학교 축구 안와골절 사고)·별첨2(음식점 가스배관 사고) 보고서 정독, 사건 흐름과 등장 문서 추출
  • 손해사정 용어집(Glossary) 초안 작성 — 사고/사건/케이스/청구/종결 등 동의어 통일
  • 보험업법 제185조, 시행세칙(2025.10.28), 시행령 일부개정안(2026.1.1) 최신 문구 확보
  • 표준 손해사정서 양식 1~2종 확보 (PDF/DOCX)
  • 현장에서 활동 중인 사정사 3~5인 인터뷰 일정 잡기
산출물
도메인 용어집 v0.1, 사건 흐름 메모, 법령 발췌본, 양식 원본
합격선
별첨1·2의 모든 등장 문서·등장 행위자·시간 순서를 글로 설명할 수 있다.
1주차

사정사 인터뷰 + 사건 흐름 워크숍

소요 1주
활동
  • 사정사 3~5인 인터뷰 — 종결 시 손해사정서 작성·교부·인증의 실제 흐름, 보험사별 차이, 가장 시간이 많이 드는 부분, 거부감의 원인
  • 사건 흐름 워크숍 — 사건을 시간 순서로 나열, 도메인 묶음과 경계 식별 (예: "사건", "당사자", "서류", "보고서", "교부 기록", "동의" 등)
  • "out of scope" 1차 결정 — 1단계가 다루지 않을 것 명시 (보고서 자동 작성, 중복보험 안분, 피해자 매칭, 보험내역 연계 등)
산출물
인터뷰 노트, 도메인 사건 지도(이벤트 스토밍 결과), 도메인 묶음 목록, 범위 밖 목록
합격선
한 사건이 어플 안에서 어떤 사건들을 거치는지 도식으로 그릴 수 있다. 도메인 묶음이 5~8개로 정리됐다.
2주차

위험 식별과 예광탄 검증

소요 1주
활동
  • 위험 목록 작성 — 보험사 교부 기록 채널, 전자서명 법적 효력, 민감정보 암호화, PDF 자동 작성 정확도, 사정사 디지털 적응
  • 예광탄 1편: "빈 손해사정서 → PDF 자동 작성 → 가짜 보험사 메일함으로 송부 → 수신 시각 기록"을 4~5일 안에 한 번 흘려보낸다 (UI 없이 명령줄로도 OK)
  • 예광탄 2편: 전자서명 라이브러리 1~2종 후보를 골라 정자체 사인 자동 기입 + 위치 사전 지정 가능 여부 검증
산출물
위험 매트릭스(영향/확률/대응), 예광탄 코드 2편, 라이브러리 후보 비교 메모
합격선
"이 흐름이 작동하는가"라는 의문 5개 중 최소 3개가 해소됐다. 작동 안 하는 부분은 대안이 있다.
3주차

기술 결정 문서(RFC) 작성과 동료 리뷰

소요 3~5일
활동
  • RFC 1편 작성 — 1단계 첫 출시판의 목표·아키텍처·도메인 모델 요약·외부 의존·대안·결정 사항·범위 밖
  • 구체 결정: PDF 라이브러리 선택, 전자서명 방식, 교부 기록 채널 1순위·2순위, 민감정보 암호화 방식, 인증 방식, 배포 환경
  • 동료 리뷰 — 적어도 1~2인이 글로 의견 남기게 한다. 결정 사항을 본인 글로 다시 쓸 수 있어야 함
산출물
docs/rfc/001-compliance-mvp.md
합격선
RFC를 처음 보는 사람이 30분 안에 "이 어플의 1단계가 뭘 하는지·어떻게 만드는지"를 답할 수 있다.
4주차

뼈대 시스템 배포

소요 1주
활동
  • 저장소 구조 결정 — backend/(Spring Boot), frontend/(Next.js), spec/, docs/
  • 최소 흐름 1개를 끝과 끝으로 연결: "사정사 로그인 → 빈 사건 1건 생성 → DB 저장 → 화면 표시"
  • Docker 이미지 빌드, MySQL 컨테이너 연결, 개발 환경 배포
  • 지속적 통합(CI) — 푸시 시 빌드·테스트·정적 분석 자동 실행
  • 로그·기본 모니터링 한 줄짜리 첫 설치
산출물
운영 환경에 배포된 빈 어플, CI 설정, README와 실행 가이드
합격선
새 개발자가 30분 안에 로컬에서 어플을 띄우고 1건 입력해 볼 수 있다. 푸시 한 번에 자동 배포된다.
5주차

첫 스펙 작성으로 진입

소요 3~5일
활동
  • spec/ 폴더 골격 잡기 — 스펙은 도메인 묶음 단위로 1개 파일 1~3쪽
  • EARS 패턴으로 1단계 첫 출시판 스펙 작성 (손해사정서 자동 작성, 피해자 전자 확인, 교부 기록)
  • 각 스펙마다 "out of scope" 절·인수 시나리오·외부 의존 명시
  • 스펙 → 계획 → 작업 → 구현 흐름 시작
산출물
spec/ 폴더의 3~5편 스펙
합격선
스펙 하나를 읽고 인수 테스트 3건 이상을 자동으로 도출할 수 있다.

전체 일정 한눈에

  • 0주차 (1주)도메인 학습 — 별첨 정독·용어집·법령·양식 수집
  • 1주차 (1주)사정사 인터뷰 + 사건 흐름 워크숍 + 범위 밖 결정
  • 2주차 (1주)위험 매트릭스 + 예광탄 2편 (PDF·전자서명·교부 기록)
  • 3주차 (3~5일)RFC 작성 + 동료 리뷰
  • 4주차 (1주)뼈대 시스템 운영 배포 + CI 설정
  • 5주차 (3~5일)스펙 작성 진입 — 1단계 첫 출시판 본격 개발 시작
  • 합계약 4~6주 (소속/일정에 따라 압축 가능)

4. 단계별 산출물 체크리스트

0주차 — 도메인 학습

  • 도메인 용어집 v0.1 (docs/glossary.md) — 사정사 용어 30개 이상 정의
  • 별첨1·2 사건 흐름 요약 메모
  • 법령 발췌본 (제185조·시행세칙·시행령 일부개정안)
  • 표준 손해사정서 양식 1종 이상 PDF/DOCX 확보
  • 사정사 인터뷰 일정 3명 이상 확정

1주차 — 인터뷰 + 사건 흐름

  • 인터뷰 노트 3편 이상
  • 도메인 사건 지도 (이벤트 스토밍 결과를 사진/도식으로 보존)
  • 도메인 묶음 목록 5~8개
  • 1단계 범위 밖 목록 (안 만들 기능 명시)

2주차 — 위험 + 예광탄

  • 위험 매트릭스 (영향 × 확률 × 대응)
  • 예광탄 1편 코드 — PDF 자동 작성 + 교부 송부 + 수신 기록
  • 예광탄 2편 코드 — 전자서명/정자체 사인 자동 기입
  • 라이브러리 후보 비교 메모

3주차 — RFC

  • docs/rfc/001-compliance-mvp.md
  • 아키텍처 도식 (단순 사각형/화살표 수준이라도)
  • 1~2인 동료 리뷰 코멘트 반영

4주차 — 뼈대 시스템

  • 저장소 골격 (backend/, frontend/, spec/, docs/)
  • Docker Compose 또는 개발 환경 실행 스크립트
  • CI 워크플로 (빌드·테스트·정적 분석)
  • 개발 환경 배포 1회 성공
  • README — 30분 안에 띄우는 방법

5주차 — 스펙 진입

  • spec/ 폴더 3~5편
  • 각 스펙은 1~3쪽, 범위 밖 명시, 인수 시나리오 포함
  • EARS 패턴 적용
  • 도메인 용어집과 스펙 단어 교차 검증

5. 피해야 할 함정(안티 패턴)

함정 1 — 사전 단계 없이 바로 스펙

현상: "그냥 적자"고 시작하면 도메인 용어가 흔들리고, "사건"인지 "케이스"인지 "청구"인지가 절마다 다르다.

대응: 1주차에 도메인 용어집을 먼저 굳힌다. 같은 개념을 하나의 단어로만 부른다.

함정 2 — 사전 단계가 끝나지 않고 무한 연장

현상: 인터뷰가 인터뷰를 부르고, RFC가 RFC를 부른다. 코드가 한 줄도 안 나온다.

대응: 각 단계에 합격선과 마감을 둔다. 6주가 지나면 무조건 스펙으로 넘어간다. 부족한 부분은 스펙 진입 후에도 보강 가능하다.

함정 3 — 뼈대 시스템을 마지막으로 미룸

현상: 기능을 다 만든 뒤에 배포하려고 하면 빌드·CI·인증·로그·환경 변수 같은 횡단 이슈가 한꺼번에 터진다.

대응: 4주차에 무조건 운영 환경에 빈 어플을 한 번 띄운다. 이후 모든 기능은 이 뼈대 위에 살을 붙이는 방식.

함정 4 — 예광탄을 프로토타입처럼 버림

현상: 예광탄으로 검증한 코드를 "임시"라며 버리고 다시 짠다. 같은 위험을 두 번 검증한다.

대응: 예광탄은 본 시스템의 시작점이다. 코드 품질을 처음부터 운영 수준으로 유지한다. 프로토타입과 예광탄은 다른 개념.

함정 5 — 첫 스펙이 너무 김

현상: 한 스펙에 모든 기능을 욱여넣어 10쪽이 넘는다. 안 읽힌다.

대응: 1~3쪽을 넘으면 쪼갠다. 도메인 묶음 단위로 스펙 1편.

함정 6 — "out of scope"를 적지 않음

현상: 구현 중에 "이것도 1단계 아닌가?"가 계속 나온다. 범위가 부푼다.

대응: 모든 스펙은 "이 스펙이 다루지 않는 것" 절을 갖는다. 명시되지 않은 것은 다음 단계로.

6. 스펙 작성으로 넘어가는 시점 — 합격선

다음 6개가 모두 yes면 5주차의 스펙 작성을 시작한다. 하나라도 no면 그 항목을 보강한 뒤 진입한다.
