한 줄 결론

tmux를 "AI 에이전트 회사의 운영체제"로 사용한다. 사용자는 감독자(CEO/오너) 패널 1개를 점유하고, 역할별 에이전트(PM·아키텍트·백엔드·프론트엔드·QA·DevOps·보안)를 각자의 패널에 띄운다. 각 에이전트는 자기 전용 git 워크트리(작업 분기)에서 일하고, 작업이 끝나면 PR을 올려 감독자 검토를 받는다. 사용자는 send-keys로 지시하고 capture-pane으로 상태를 본다. 야간·장시간 작업은 랠프 루프로 자율 반복.

왜 tmux인가. 프로세스 격리·세션 영속성·실시간 출력·원격 접속이라는 tmux의 본래 특성이 AI 에이전트 오케스트레이션이 필요로 하는 속성과 그대로 일치한다. 별도 플랫폼을 만들 필요 없이 2007년부터 존재한 도구가 이미 정답이다.

1. 왜 tmux + 멀티 AI 에이전트인가

tmux의 본래 특성이 AI 에이전트 운영의 요구와 정확히 일치한다.

tmux 본래 특성 AI 에이전트 운영에서의 의미
패널 격리 각 에이전트가 다른 에이전트의 출력에 간섭하지 않음. 한 명이 죽어도 다른 사람 일은 계속됨.
세션 영속성 터미널을 닫아도 에이전트들이 계속 일함. 다음 날 다시 붙으면 그대로 이어 일하고 있음.
실시간 출력 감독자가 어느 에이전트가 무엇을 하고 있는지 한눈에 봄.
send-keys / capture-pane 에이전트 사이 통신 메커니즘 그대로 사용 가능. 감독자가 "타이핑해 주는 사람"처럼 지시 전달.
원격 접속(SSH) 외부에서 같은 세션에 붙어 상태 확인. 모바일에서 진행 상황 모니터링 가능.
스크립트화 레이아웃·시작 명령·역할 부여를 한 줄 스크립트로 재현. 회사 셋업을 코드로 관리.

현실적 규모

실무 사례 기준 3~5명으로 시작해 10~20명까지 확장이 일반적이다. 16GB RAM 맥에서 20명 동시 가동 무리 없음. 그러나 현실 제약은 메모리가 아니라 검토 대역폭이다. 한 사람이 동시에 검토할 수 있는 병렬 PR은 5~8개가 한계. 처음에는 3~5명으로 좁히는 것을 권장.

2. 업계 패턴 6가지

1 감독자–작업자 계층(Supervisor–Worker)

핵심
최상위 감독자가 작업을 분해해 작업자 에이전트에게 분배한다. 사무라이 비유로 "쇼군 → 카로 → 아시가루" 같은 3단 계층도 사례에 있다.
왜 필요
한 에이전트가 모든 결정을 내리면 컨텍스트 창이 터지고, 책임이 흐려진다. 계층화는 책임 경계를 명확히 한다.
우리 적용
감독자(사용자) → PM(스펙 작성·검수) → SWE(구현) + QA(검증) 구조. 필요 시 아키텍트를 별도 계층으로 분리.

2 워크트리 격리(Worktree Isolation)

핵심
git worktree는 하나의 저장소에 여러 작업 디렉터리를 만들어 각자 다른 브랜치에서 동시에 작업하게 한다. 에이전트별로 워크트리 하나·브랜치 하나·PR 하나.
왜 필요
같은 디렉터리에서 두 에이전트가 동시에 파일을 수정하면 충돌·덮어쓰기·꼬임이 생긴다. 워크트리는 디스크 수준에서 분리한다.
우리 적용
main 외에 backend/feat-pdf-generator, frontend/feat-sign-flow, qa/integration-tests 같은 워크트리를 동시에 띄움.

3 역할 기반 에이전트(Role-Based Agents)

핵심
PM·아키텍트·백엔드·프론트엔드·QA·DevOps·보안 등 역할별로 별도 에이전트. 각 에이전트는 자기 시스템 프롬프트(역할 설명·금지 행동·도구 제한)를 가짐.
왜 필요
"만능 에이전트"는 결정이 느리고 출력 품질이 평균값에 가깝다. 역할로 좁히면 깊이가 생긴다.
우리 적용
역할별 시스템 프롬프트를 .claude/agents/ 또는 동등 위치에 저장. 우리 도메인 용어집과 법령 발췌본을 보안·PM 에이전트에 주입.

4 감독 루프(PM → SWE → QA → PM)

핵심
PM이 작업을 명세화 → SWE가 구현 → QA가 인수 기준 통과 여부 검증 → PM이 사용자 관점에서 최종 검수. 통과 못하면 다시 SWE로.
왜 필요
한 에이전트가 자기 코드를 자기가 검증하면 "다 됐다"가 거짓일 가능성이 높다. 역할이 다른 에이전트가 검증해야 신뢰가 생긴다.
우리 적용
PR이 열리면 자동으로 QA 에이전트가 인수 시나리오 실행. 통과 후 PM 에이전트가 스펙 정합성 검토. 둘 다 통과해야 감독자에게 알림.

5 랠프 루프(Ralph Loop) — 자율 반복

핵심
"제품 요구 문서(PRD)의 모든 항목이 완료될 때까지" 매 반복마다 새 컨텍스트로 에이전트를 다시 띄워 다음 항목을 처리한다. 진행 상태는 파일과 git 이력에 남기고, LLM 컨텍스트에는 남기지 않는다.
왜 필요
긴 작업에서 컨텍스트 창이 가득 차는 문제를 해결. 사용자가 잠든 사이에도 작업이 진행됨.
우리 적용
1단계 첫 출시판의 인수 시나리오 30~50개를 PRD로 작성 → 랠프 루프가 야간에 하나씩 통과시킴 → 아침에 통과 결과 검토.

6 패널 간 통신 — send-keys와 capture-pane

핵심
tmux send-keys는 다른 패널의 입력창에 "사람이 친 것처럼" 텍스트를 주입한다. tmux capture-pane은 다른 패널의 화면을 캡처해 읽는다. 이 두 명령만으로 에이전트 사이 통신이 가능하다.
왜 필요
별도의 메시지 큐·API 없이도 에이전트들이 협업할 수 있다. 가장 단순하고 깨지지 않는 통신 방식.
우리 적용
감독자 스크립트가 "QA야, 이 PR 검증해 줘"를 QA 패널에 send-keys로 주입. 결과가 화면에 나오면 capture-pane으로 읽어 다음 단계로 넘김.

워크트리 디스크 구조

~/IdeaProjects/Adjuster-system/         # 메인 저장소 (main 브랜치)
~/work/adjuster-pm/                     # PM 에이전트 (spec/ 작성)
~/work/adjuster-backend-pdf/            # Backend, feat-pdf-generator 브랜치
~/work/adjuster-frontend-sign/          # Frontend, feat-sign-flow 브랜치
~/work/adjuster-qa/                     # QA, qa/integration-tests 브랜치
~/work/adjuster-compliance/             # Compliance, audit/...

각 워크트리는 같은 저장소를 공유하므로 git fetch 한 번이면 다른 에이전트의 변경 사항을 가져올 수 있다. 작업이 끝나면 PR로 main에 합친다.

tmux 셋업 명령 예시

# 새 세션
tmux new-session -d -s adjuster-coop -n main

# CEO 패널은 첫 패널 그대로

# PM 패널
tmux split-window -h -t adjuster-coop:main
tmux send-keys -t adjuster-coop:main.1 \
  "cd ~/work/adjuster-pm && claude --agent pm" C-m

# Backend 패널
tmux split-window -v -t adjuster-coop:main.1
tmux send-keys -t adjuster-coop:main.2 \
  "cd ~/work/adjuster-backend-pdf && claude --agent backend" C-m

# (Frontend, QA, Compliance도 동일 방식)

tmux attach -t adjuster-coop

실제로는 이 셋업을 scripts/launch-team.sh로 만들어 한 줄로 회사를 띄운다.

5. 핵심 워크플로 — PM → SWE → QA 루프

PM
스펙 작성
→
Architect
RFC 검토
→
SWE
구현 + PR
→
QA
인수 시나리오
→
PM
최종 검수
↻

한 작업의 일생

  1. 감독자가 작업 지시 — "1단계 스펙 #03: 손해사정서 PDF 자동 생성을 구현"
  2. PM 에이전트가 스펙 확정 — spec/03-pdf-generator.md에 EARS 문장과 인수 시나리오 5건 작성. send-keys로 감독자에게 "검토 부탁드립니다" 통지.
  3. 감독자가 스펙 승인 또는 수정 요청 — 감독자 패널에서 한 줄로 응답.
  4. Architect 에이전트가 RFC 짧게 검토 — "PDF 라이브러리는 OpenPDF + Spring Boot 통합으로. 폰트 임베딩 주의."
  5. Backend 에이전트가 자기 워크트리에서 구현 — feat-pdf-generator 브랜치. 완료 시 PR 생성.
  6. QA 에이전트가 PR을 자기 워크트리로 체크아웃해 인수 시나리오 실행 — 통과/실패 결과를 PR 코멘트로 남김. send-keys로 PM에게 알림.
  7. PM 에이전트가 스펙 정합성 최종 검수 — 통과면 PR 승인 의견. 실패면 send-keys로 Backend에 재작업 요청.
  8. 감독자가 머지 — 최종 결정. 머지 후 scripts/sync-worktrees.sh로 모든 에이전트 워크트리 동기화.

중요 원칙 — 코드는 한 에이전트씩

여러 에이전트가 같은 파일을 동시에 수정하면 머지 충돌이 즉시 발생한다. 한 워크트리 = 한 작업 = 한 PR을 엄격히 지킨다. 작업이 겹치면 PM 에이전트가 분해해 순서를 정한다.

6. 랠프 루프(Ralph Loop) 통합 — 야간 자율 개발

장시간·반복 작업은 랠프 루프에 맡긴다. 동작 원리는 단순하다.

  1. PRD 파일에 남은 작업 목록을 기록한다 — docs/prd/phase-1-todo.md.
  2. 루프가 다음 작업을 골라 새 컨텍스트로 에이전트를 띄운다.
  3. 에이전트가 작업을 완료하고 PRD를 갱신한다 — "[x] AC #12 통과".
  4. QA·테스트가 자동으로 검증한다. 실패면 다음 반복에서 같은 작업 재시도.
  5. 모든 PRD 항목이 [x]가 되면 루프 종료.

이 환경에는 이미 ralph-loop 스킬이 설치돼 있다(/ralph-loop). 다음 형태로 호출:

/ralph-loop docs/prd/phase-1-todo.md

잘 동작하는 조건

  • 각 PRD 항목이 자동 검증 가능해야 함 (테스트·타입체커·린터)
  • 각 항목의 범위가 한 작업 파일에 담길 정도로 좁아야 함
  • 신규 구현(greenfield)에 강함. 기존 코드 대규모 리팩토링은 약함

따라서 1단계 첫 출시판처럼 신규로 만드는 어플에는 잘 맞는다.

7. 도구 후보 비교

2026년 5월 기준, tmux 멀티 에이전트 오케스트레이터는 빠르게 늘고 있다. 우리 요구(1인 개발자, Claude Code 기반, 무료 또는 저비용)에 맞춰 비교한다.

도구 특징 적합도
tmux + Claude Code (수제) tmux 명령과 Claude Code 하위 에이전트 + 워크트리만 조합. 외부 도구 의존 없음. 셋업 스크립트만 손수 작성. 가장 권장. 학습 비용 낮고 환경 의존성 0.
agentmux tmux 기반 빠른 오케스트레이터. macOS·Linux·WSL. 패널 레이아웃 자동. 수제 셋업이 번거로워질 때 검토.
ittybitty Claude 인스턴스를 tmux 가상 터미널에 자동 띄움. 한 에이전트가 다른 에이전트를 또 띄울 수 있음. 계층적 위임이 필요하면 검토.
CLI Agent Orchestrator (CAO) AWS Labs 오픈소스. 다양한 CLI 에이전트(Claude·Codex·Gemini 등) 통합. MCP 기반 감독자-작업자. 여러 모델을 섞어 쓸 계획이면 검토.
Tmux-Orchestrator (absmartly) tmux + Claude 에이전트 관리 오픈소스. 셋업 스크립트 다수 포함. 수제 셋업 참고용으로 가치.
Agent Orchestrator (Composio) 병렬 코딩 에이전트. 각자 워크트리·브랜치·PR. CI 실패 자동 수정·머지 충돌 처리. 여러 PR을 동시에 굴리는 단계 이후 검토.
ralph (snarktank) 자율 반복 루프 원조 구현. PRD 모두 완료될 때까지. 야간 자율 개발 도입 시 채택. 본 환경엔 ralph-loop 스킬로 이미 사용 가능.

1차 권고

"수제 tmux + Claude Code 하위 에이전트 + git 워크트리 + 랠프 루프" 조합으로 시작. 셋업 스크립트 한두 개만 손수 작성하면 외부 도구 의존 없이 완성된다. 운영하면서 부족함이 명확해지면 agentmux나 CAO 같은 전용 오케스트레이터로 옮긴다.

9. 도입 3단계 — 걷고, 뛰고, 날기

1단계 · 걷기

2명 회사 — 감독자 + 단일 Claude 에이전트

1~2일
목표

활동
  • tmux 세션 셋업, 단축키 학습
  • Claude Code 패널에 작업 하나 시켜 끝까지 따라가 본다

1주
목표

상시 운영
목표
Architect·Compliance·DevOps 추가. 야간 자율 개발 시작.

10. 안티패턴

함정 1 — 한 에이전트에 모든 역할 떠넘기기

현상: "이 Claude한테 다 시키자"고 하면 컨텍스트 창이 곧 가득 차고, 자기 코드를 자기가 검증해 거짓 통과가 나온다.

대응: 처음부터 PM·SWE·QA를 분리. 단일 에이전트로 시작하더라도 역할 의식을 분리.

함정 2 — 같은 디렉터리에서 여러 에이전트 동시 수정

현상: 워크트리 없이 같은 폴더에서 두 에이전트가 파일을 만지면 덮어쓰기·머지 충돌이 즉시 발생.

대응: 에이전트별 워크트리를 무조건 분리. 한 워크트리 = 한 작업.

함정 3 — 검토 대역폭 초과

현상: 10명을 동시에 띄웠는데 PR이 한 번에 10개 올라온다. 감독자가 못 따라간다. 결국 대충 머지하다 사고가 난다.

대응: 처음 1개월은 3~5명만 띄움. 검토 대역폭을 우선 측정. PR 자동 검증(QA 에이전트 + CI)을 먼저 단단히 한 뒤 인원을 늘림.

함정 4 — 랠프 루프를 검증 없이 야간에 풀어 둠

현상: 인수 시나리오가 부실하면 밤사이 "통과했다"고 자기 보고된 PR이 잘못된 구현일 수 있다.

대응: 랠프 루프에 들어가는 PRD 항목은 자동 검증이 가능한 것만. 인수 시나리오는 사람이 한 번 더 본 뒤 PRD에 추가.

함정 5 — 에이전트 통신을 직접 메시지 큐로 만들려고 함

현상: 처음부터 Redis/카프카 같은 메시지 큐를 도입해 에이전트 간 통신을 구축하려다 본업이 멈춘다.

대응send-keys와 파일 시스템(공유 PRD·로그 파일)만으로 충분히 멀리 갈 수 있다. 메시지 큐는 마지막 수단.

함정 6 — 도구를 너무 일찍 갈아탐

현상: 새 오케스트레이터가 매주 나온다. 매주 셋업을 다시 한다.

대응: 1차로 정한 조합(수제 tmux + Claude Code + 워크트리 + 랠프)으로 최소 1개월 운영한 뒤에만 도구 변경 검토.
